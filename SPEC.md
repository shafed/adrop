# AirDrop-подобное приложение для Arch Linux ↔ Android

## 1. Обзор и цель

Приложение для быстрой передачи **файлов** и **содержимого буфера обмена**
между ПК на Arch Linux и телефоном на Android — по аналогии с AirDrop.

Основной сценарий: я отправляю файлы/буфер с одного устройства, и они
появляются на другом без лишних действий. Передача идёт напрямую между двумя
моими устройствами (peer-to-peer), доверенными после однократного сопряжения.

Два устройства считаются **моими собственными**, поэтому после сопряжения
ручное подтверждение каждой передачи не требуется — приоритет отдан скорости и
отсутствию трения.

---

## 2. Архитектура (общая)

```
┌─────────────────── Arch Linux (ПК) ───────────────────┐        ┌──────────── Android ────────────┐
│                                                        │        │                                 │
│  adrop CLI  ──(unix socket)──►  adrop daemon           │        │   Native Kotlin app             │
│  (тонкий клиент)                (резидентный процесс)   │        │   (Jetpack Compose)             │
│                                  │                      │        │                                 │
│                                  └──── pinned-TLS ──────┼────────┼──► foreground-сервис приёма     │
│                                       (LAN, P2P)        │  LAN   │     (только в "окне приёма")    │
└────────────────────────────────────────────────────────┘        └─────────────────────────────────┘
```

- **ПК-сторона:** фоновый демон (`adrop daemon`) + тонкий CLI (`adrop`),
  общающиеся через Unix-domain socket. Демон держит состояние сопряжения,
  слушает входящие соединения, показывает уведомления, работает с буфером.
- **Android-сторона:** нативное приложение на Kotlin/Compose. Принимает данные
  через foreground-сервис, который поднимается **только на время окна приёма**
  (а не постоянно), ради экономии батареи.
- **Транспорт:** прямое P2P-соединение по LAN поверх взаимно
  аутентифицированного TLS с пиннингом ключей, полученных при сопряжении.

## 3. Платформенные допущения

| | |
|---|---|
| ПК ОС | Arch Linux |
| ПК DE/дисплей | Wayland (KDE Plasma / Sway / Hyprland) |
| ПК стек | **Go** (один статический бинарь, демон + CLI) |
| Буфер обмена ПК | `wl-clipboard` (`wl-copy` / `wl-paste`) |
| Уведомления ПК | libnotify / `notify-send` (через D-Bus уведомлений) |
| Android | Нативное приложение, Kotlin + Jetpack Compose |
| Android приём файлов | MediaStore → общий каталог Downloads (без широких прав на хранилище) |

## 4. Обнаружение и соединение

- **Без авто-обнаружения (mDNS).** Используется **ручное сопряжение** +
  прямое соединение по IP/хосту.
- При сопряжении устройства обмениваются публичными ключами и сетевым адресом.
- Соединение устанавливается напрямую по сохранённому адресу в пределах LAN.

### Сопряжение (pairing)

- **ПК показывает QR-код, телефон сканирует камерой.**
- QR кодирует: публичный ключ ПК + сетевой адрес (+ идентификатор устройства).
- QR может рендериться в терминале (CLI) и/или в окне.
- После сканирования стороны обмениваются ключами и сохраняют друг друга как
  доверенные устройства.

### Модель доверия

- **Сопряжение один раз → дальнейшие передачи между доверенными устройствами
  принимаются автоматически** (без accept/reject на каждую передачу).

---

## 5. Безопасность

- **TLS со пиннингом ключей сопряжения.**
- Каждое соединение — взаимно аутентифицированный TLS, привязанный к ключам,
  полученным при сопряжении (защита от MITM, без паролей).
- Контент (файлы, буфер) никогда не проходит через облако в режиме MVP
  (всё P2P по LAN).

---

## 6. Управление устройствами

- **Несколько устройств, с возможностью отзыва.**
- Хранится список сопряжённых устройств: имя + пиннингованный публичный ключ.
- Можно сопрягать больше двух устройств (на будущее: планшет, второй ПК).
- Любое устройство можно **отозвать** из списка (revoke).

---

## 7. Передача файлов

### Множественная отправка

- CLI принимает несколько путей: `adrop send file1.pdf photo.jpg notes.txt`.
- Поддержка glob/wildcards (раскрываются шеллом), папок, stdin (`-`) — по мере
  реализации; базово — список файлов.
- **Группировка: одна сессия передачи.** Все файлы из одной команды передаются
  как единая сессия → на телефоне одно уведомление («Получено 3 файла»), один
  общий прогресс, файлы приходят вместе.

### Целостность и прерывания

- **Проверка хешем, без resume.**
- Для каждого файла передаётся контрольная сумма (SHA-256); получатель
  проверяет и отбрасывает при несовпадении.
- Прерванная передача начинается заново с нуля (без докачки). Приемлемо для LAN.

### Куда падают файлы

**На ПК:**
- Каталог `~/Downloads`.
- При коллизии имён — автопереименование: `file.pdf` → `file (1).pdf` и т.д.
- Никогда не перезаписывает, никогда не спрашивает.

**На Android:**
- Сохранение в общий каталог Downloads через **MediaStore**
  (не требует широкого разрешения на хранилище на современных Android).
- Файлы видны другим приложениям/галерее.

---

## 8. Поведение получателя (приём)

- **Автоприём + уведомление, без подтверждения.**
- При входящей передаче приложение/демон принимает данные автоматически и
  показывает **уведомление** (напр. «Получен report.pdf»).
- Приложение **не** всплывает в foreground и **не** показывает кнопок
  accept/reject. Нажатие на уведомление открывает/показывает файл.
- Это сознательное уточнение исходной формулировки SPEC («приложение
  открывается для подтверждения»): поскольку устройства мои собственные,
  выбран автоприём с уведомлением вместо подтверждения.

---

## 9. Буфер обмена

- **Только ручная отправка (manual push).**
- Я явно инициирую «отправить буфер» с одного устройства; ничего не происходит
  автоматически и нет фонового мониторинга буфера.
- На стороне-получателе содержимое **молча устанавливается в системный буфер**
  (плюс уведомление) — чтобы сразу можно было вставить.
  - На ПК: запись через `wl-copy`.
- Формат содержимого уточняется при реализации (как минимум — текст).

---

## 10. Жизненный цикл и IPC на ПК

- **IPC daemon ↔ CLI:** Unix-domain socket
  (напр. `$XDG_RUNTIME_DIR/adrop.sock`). Безопасно (права ФС), без порта.
- **Запуск демона:** systemd **user**-сервис.
  - Поставляется unit-файл; `systemctl --user enable --now adrop`.
  - Автозапуск при логине, логи через journald, рестарт при падении.

## 11. MVP (первая «ежедневно пригодная» версия)

**В объёме MVP:**
1. Сопряжение (QR на ПК → скан телефоном), обмен ключами, хранение
   доверенных устройств (несколько, с отзывом).
2. Двунаправленная передача **файлов** поверх pinned-TLS по **LAN**.
   - Множественная отправка одной сессией, проверка SHA-256.
   - Падение в `~/Downloads` (ПК, автопереименование) и Downloads/MediaStore
     (Android).
3. Двунаправленная **ручная отправка буфера обмена** (как минимум текст),
   молчаливая установка в системный буфер на приёмнике.
4. ПК: демон (systemd user) + CLI (`adrop send ...`) через Unix socket.
5. Android: приём через foreground-сервис, поднимаемый **на время окна приёма**.
6. Автоприём + уведомления на обеих сторонах.

**Явно отложено в Phase 2 (см. ниже).**

## 12. Phase 2 (после MVP)

### 12.1 Пробуждение телефона через FCM
Так как телефон слушает только в «окне приёма», закрытый телефон не может
принять push с ПК. Решение на phase 2:

- **FCM-пробуждение:** ПК → небольшой relay → Firebase Cloud Messaging
  «будит» приложение, открывая окно приёма автоматически.
- **Через relay/облако проходит ТОЛЬКО сигнал пробуждения** («проснись, пир X
  хочет отправить»). Сам контент (файлы, буфер) после этого идёт напрямую
  device-to-device по pinned-TLS LAN. **Облако контента не видит.**
- Требует: Firebase-проект + минимальный content-free relay-эндпоинт.

### 12.2 Возможные расширения
- Resume/докачка прерванных передач (чанки).
- Передача папок с сохранением структуры / авто-архивация больших пачек.
- Буфер обмена: изображения и rich-форматы помимо текста.
- mDNS-автообнаружение как дополнение к ручному сопряжению.
- Relay-fallback для передачи вне общей LAN (E2E-шифрование, relay не читает).
- Per-file прогресс внутри сессии.
- Интеграция в файловый менеджер (KDE Dolphin «Отправить на телефон»).

## 13. Открытые вопросы / решить при реализации

- Точный формат содержимого буфера для MVP (plain text vs +HTML/RTF).
- Поведение CLI при stdin/pipe и при отправке каталогов (рекурсия vs архив).
- Имя/брендинг бинаря и пакета (`adrop` — рабочее название).
- Формат и версия протокола поверх TLS (framing, метаданные сессии).
- Конкретный механизм «окна приёма» на Android (как и насколько долго держится
  foreground-сервис; как закрывается).
